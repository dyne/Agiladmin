#+TITLE: Incanter Removal Migration Plan
#+AUTHOR: Codex

* Summary

The real goal is to remove the Incanter dependency while preserving
behavior.

The easiest path is *not* ~core.matrix~. The easiest path is a local
tabular layer built on plain Clojure data:

#+begin_src clojure
{:column-names [:month :name :project :task :tag :hours]
 :rows [{:month "2026-01" :name "Alice" ...}
        ...]}
#+end_src

with transforms implemented via:

- ~mapv~
- ~filterv~
- ~reduce~
- ~group-by~
- ~sort-by~
- ~select-keys~
- ~dissoc~
- small local helper functions

If the goal is "remove Incanter without changing behavior", the safest
path is:

1. isolate all Incanter usage behind a local tabular facade
2. replace call sites with that facade
3. switch the facade implementation to plain seqs/maps
4. remove Incanter from ~deps.edn~

* Difficulty Assessment

** Overall

- *Direct substitution with no intermediate facade:* hard
- *Facade-first migration with behavior locking tests:* medium-hard
- *If willing to replace Incanter datasets with plain seq/map transforms:* medium
- *If insisting on core.matrix as the main tabular representation:* hard

** Why it is hard here

The repo depends on Incanter dataset features in multiple places:

- dataset construction
- column projection and exclusion
- row filtering
- grouping and aggregation
- derived columns
- row ordering
- row/column concatenation
- dataset-to-list conversion for rendering/export

These are not the strengths of ~core.matrix~.

* Coordinate System Concern

Your concern about different XY defaults is valid in general, but in
this repository it is probably *not the dominant cost*.

Why:

- the current graphics namespace does not actively use ~incanter.charts~
  in production code
- most rendering is HTML tables and derived progress values
- the project Gantt view is driven by JS data structures, not Incanter
  charts

So the likely migration pain is not "x/y inversion" but:

- losing dataset semantics
- replacing column-oriented transforms
- preserving ordering and rendering shape
- keeping numeric output identical enough for tests and reports

* Recommendation

Do *not* replace Incanter with raw ~core.matrix~ directly.

The recommended target for this repository is:

1. a local plain-data tabular facade
2. vectors of row maps as the backing representation
3. no new external dataframe dependency unless the plain-data approach
   proves insufficient

** Phase 3: Replace Incanter-backed facade implementation

Options:

*** Option A: plain Clojure data

Best for preserving behavior with lowest conceptual overhead.

Use:

- vectors of row maps
- ~group-by~, ~reduce~, ~sort-by~, ~mapv~, ~filterv~

Pros:

- smallest dependency surface
- easiest to debug
- best fit for this repo's current usage
- matches current tests well

Cons:

- more custom code than a dataframe library

*** Option B: core.matrix-backed numeric operations only

Use ~core.matrix~ only where true matrix operations help.

Pros:

- keeps numeric kernels available

Cons:

- still requires writing a custom dataset layer
- does not reduce most migration work

*** Option C: core.matrix as primary table layer

Not recommended.

Pros:

- single target library

Cons:

- poor fit for heterogeneous row maps and named columns
- large adapter layer required
- highest behavior-regression risk

Exit criteria:

- facade no longer depends on Incanter
- tests remain green

* Suggested Next Implementation Slices

The first slice is already done: create ~agiladmin.tabular~ and migrate
low-risk consumers.

Recommended next slices:

1. identify any residual direct Incanter calls outside the migrated slices
2. migrate tests from ~inc/to-dataset~ fixtures to [[/home/jrml/devel/planb-agiladmin/src/agiladmin/tabular.clj][src/agiladmin/tabular.clj]]
3. migrate any remaining aggregation helpers not yet routed through the tabular facade
4. remove ~incanter/incanter-core~ only after all app and test call sites are gone

This ordering minimizes risk because it removes view-level consumption
before replacing the central transformation engine.

* Bottom Line

Removing Incanter is feasible, but the easiest route is a plain
seqs/maps tabular layer, not ~core.matrix~.

The biggest challenge is preserving dataframe-like behavior, column
order, and report output while replacing the current Incanter DSL.
