* DONE [#A] Define the authorization matrix and the enforcement seam
Effort: S
Goal: Freeze the exact role-to-route policy and decide one routing-level mechanism that all protected endpoints must use.
Notes: Keep this slice focused on policy and entry-point design. The main constraint is that denied access must happen before route handlers execute view or domain work. Do not leave policy implicit in prose only: write the exact route matrix in code comments or implementation notes before editing handlers.

** DONE [#A] Map every existing protected route to allowed roles
Why: The current routes mostly rely on generic login checks, so the exact authorization surface must be enumerated before changing behavior.
Change: Audit the routes in `src/agiladmin/handlers.clj` and classify every route explicitly. Use this matrix as the baseline and copy it into implementation notes so it cannot drift:
- Public: `GET /`, `GET /login`, `POST /login`, `GET /logout`, `GET /signup`, `POST /signup`, activation routes.
- Admin only: `GET /reload`, `GET /config`, `GET /config/edit`, `POST /config/edit`, any future operational/configuration route.
- Admin and manager: `GET /projects/list`, `POST /project`.
- Admin only: `POST /projects/edit`.
- Any authenticated role: `GET /timesheets`, `POST /timesheets/upload`, `POST /timesheets/cancel`, `POST /timesheets/submit`, `GET /timesheets/download/:path`, `GET /persons/list`, `POST /person`, `POST /persons/spreadsheet`, `POST /`.
- Special scope rules on top of role access:
  - admin may target any person and sees full project/person data including costs.
  - manager may access only own person pages from personnel routes; may list project entry points and open each project through the normal project view route; must never see or access project edit entry points; must never receive cost data or trigger cost calculation.
  - any other authenticated role may access only own person pages and upload/timesheet routes; project routes must be denied.
Document the concrete deny behavior expected for each route family: render the existing unauthorized response and do not redirect into an allowed route unless the current route already does that intentionally.
Tests: Add or update route-level tests that express the role matrix for each protected endpoint, including explicit deny cases.
Done when: There is a written route matrix covering all protected endpoints and no protected route remains on the generic "logged in" rule by accident.

** DONE [#A] Introduce a dedicated authorization layer in the session/routing seam
Why: Repeating role checks inline in route bodies will drift quickly and makes it easy to leave a route protected only in the view.
Change: Add minimal authorization helpers in `src/agiladmin/session.clj` that build on the existing normalized roles and can enforce policies before invoking route handlers. Prefer a very small fixed set of route-facing guards rather than open-ended predicates:
- one helper for authenticated-only routes,
- one helper for admin-only routes,
- one helper for manager-or-admin routes,
- one helper that resolves the effective person target for personnel routes, returning the authenticated person for non-admin users regardless of submitted params,
- one helper or capability map for `{:can-view-costs? ... :can-access-projects? ... :person-scope ...}` that handlers can pass downward.
Keep the response behavior consistent with the existing unauthorized rendering flow. Do not create a second unauthorized page style.
Tests: Add focused tests for the new session authorization helpers, covering accepted and rejected roles and preserving current behavior for unauthenticated requests.
Done when: Route definitions can express role requirements through one shared guard mechanism and no route needs to hand-roll authorization logic.

* DONE [#A] Enforce own-scope access for personnel and projects
Effort: M
Goal: Restrict non-admin users to their own data while preserving existing admin behavior.
Notes: This slice should keep the current route structure if possible. Prefer parameter validation and account-derived lookup at the route boundary over deeper view branching. Challenge any temptation to widen permissions just because a view already tolerates it. Routing decisions should produce one of three outcomes only: allow as requested, allow with account-scoped target substitution for own-person access, or deny before the handler runs.

** DONE [#A] Route non-admin personnel access only to the authenticated person
Why: The current `/persons/list` route already redirects non-admin users to their own page, but POST access to person detail still appears to rely on generic login checks.
Change: Protect the personnel routes so non-admin users can only reach their own person view, regardless of submitted parameters. For admin keep current unrestricted personnel access. For manager and other authenticated roles, derive the allowed person identity from the authenticated account at the route boundary and pass that effective identity into the view call. Be explicit per route:
- `GET /persons/list`: admin keeps the listing; every non-admin gets only own person page.
- `POST /person`: admin may request any person; every non-admin must be forced to own identity even if another person is submitted.
- `POST /persons/spreadsheet`: admin may request any valid spreadsheet export already allowed today; every non-admin must be limited to own person context or denied if the route semantics cannot safely be scoped.
If the route uses multiple parameters to identify a person or year, list them and decide which ones are account-derived and which ones remain user-provided.
Tests: Add route tests for `/persons/list` and `POST /person` showing admin access to arbitrary people, manager access only to self, and deny behavior for authenticated non-admin users targeting another person.
Done when: No non-admin route path can render another person's personnel page through crafted parameters or direct POSTs.

** DONE [#A] Define and enforce manager project scope separately from general users
Why: The request gives managers more project access than other authenticated users, so project routing needs an explicit branch instead of relying on the existing one-size-fits-all login check.
Change: Split project routing into two clearly separate policies:
- `GET /projects/list` and `POST /project`: admin and manager only.
- `POST /projects/edit`: admin only.
The policy is explicit:
- admin may list projects, open any project, and use project edit entry points with full data.
- manager may list projects entry points and open each project through `POST /project`, but must never see links/forms for `POST /projects/edit` and must be denied if that route is called directly.
- any other authenticated role must be denied on all project routes.
If the current project page mixes read-only navigation controls with edit controls, separate them so manager-visible markup contains only the read-only navigation path and admin-only markup contains the edit path.
Tests: Add route tests proving admin access, manager access only on list/detail routes, manager denial on `/projects/edit`, and denial for authenticated non-manager roles on all project routes.
Done when: Project endpoints are inaccessible to role `*`, managers reach only list/detail project surfaces, and `/projects/edit` is neither visible nor reachable for managers.
